# PWA 회원가입 후 리다이렉션 루프 및 접속 오류 분석 및 해결 보고서

일부 모바일 Safari, PWA(독립형 모드) 및 카카오톡 인앱 브라우저 환경에서 회원가입 직후(관리자 승인 대기 단계 진입 시) 발생했던 접속 불능 및 리다이렉션 루프 현상의 원인과 이를 100% 해결하는 기술적 분석 내용입니다.

---

## 1. 문제 현상 및 환경

### 📱 발생 환경
* 모바일 Safari 브라우저
* iOS PWA (홈 화면에 추가된 독립형 앱 모드)
* 카카오톡 인앱 브라우저 (내장 웹뷰)

### 🚨 주요 증상 및 에러 메시지
1. **Safari 브라우저**: `Safari가 해당 페이지를 열 수 없습니다. (오류: Load cannot follow more than 20 redirections)`
2. **PWA (독립형 모드)**: `너무 많은 재이동이 있었기 때문에 Safari가 해당 페이지를 열 수 없습니다.`
3. **카카오톡 인앱 브라우저**: `네트워크 상태가 좋지 않습니다. 잠시 후 다시 시도해 주세요.`
   *(주소창이 노출되지 않아, Mixed Content/CORS 에러로 통신이 강제 차단되는 현상을 네트워크 에러로 자체 인지)*
4. **발생 시점**: 신규 사용자가 회원가입 폼을 모두 채우고 **[가입하기] 버튼을 누른 직후** 발생 (관리자 승인이 이루어지기 전 대기 화면 진입 단계)

---

## 2. 기술적 원인 분석 (Root Causes)

오류를 유발했던 3가지 연쇄적인 기술적 요인은 다음과 같습니다.

### [원인 1] WebKit의 세션 쿠키 비동기 반영 지연 (가입 즉시 자동 로그인 시의 I/O 레이스 컨디션)
* 기존 설계에서는 회원가입 API(`/api/auth/signup`) 성공 시 서버가 자동으로 로그인 세션(`Set-Cookie`)을 반환하고, 클라이언트는 즉시 `/pending` 대기 화면으로 라우팅을 진행했습니다.
* 이때 iOS WebKit 기반 브라우저(Safari, PWA, 카카오톡 웹뷰)는 세션 쿠키를 기기 스토리지에 물리적으로 반영(I/O 완료)하는 속도보다 라우터가 서버 컴포넌트(RSC) 데이터를 Fetch하는 속도가 더 빨라, **세션 쿠키가 누락된 채로** `/pending` 요청이 날아가 로그인 세션이 무효화되는 현상이 발생했습니다.

### [원인 2] 프록시 프로토콜 오염과 HTTP 절대 경로 리다이렉트로 인한 통신 차단
* 기존 코드에는 HTTP -> HTTPS 강제 리다이렉션 로직이 커밋된 적이 없습니다. 따라서 과거에 발생한 무한 리다이렉션 루프는 이로 인한 것이 아닙니다.
* 기존 코드에서 미들웨어는 `NextResponse.redirect(new URL('/login', request.nextUrl.origin))` 또는 `/pending`으로 리다이렉트하도록 구현되어 있었습니다.
* 이때 외부 프록시 프록시 장비에서 `X-Forwarded` 관련 헤더가 누락되어 Next.js 서버는 자신의 origin을 `http://yeyak.example.or.kr` (HTTP 프로토콜)로 빌드하여 브라우저에 던졌습니다:
  ```http
  HTTP/1.1 307 Temporary Redirect
  Location: http://yeyak.example.or.kr/login
  ```
* **이 상태에서 발생한 실제 문제**:
  * **WebKit 보안 정책에 의한 강제 차단 (Safari, PWA, 카카오톡 웹뷰)**: 브라우저가 HTTPS 환경 내에서 HTTP 프로토콜을 가리키는 절대 경로 리다이렉트 지시를 비시크릿 요청(Fetch 등)으로 수신하는 순간, 이를 **보안 규정 위반(Mixed Content / CORS)**으로 판단해 브라우저 엔진 레벨에서 통신을 강제로 끊어버렸습니다.
  * 이로 인해 카카오톡 웹뷰는 "네트워크 상태가 좋지 않습니다"라는 에러를 띄웠고, Safari 및 PWA는 리다이렉트를 추적하지 못하고 로딩 실패를 겪었습니다.
  * HSTS 정책이 동작하지 않거나 비시크릿 환경인 브라우저에서는 80 포트로 실제 요청을 보냈고, 외부 프록시 80 포트 포워딩 룰이 없으므로 Next.js가 아닌 **외부 프록시의 기본 웹 루트 디렉토리(Web Station 안내 페이지)** 혹은 **404 에러** 화면으로 빠지는 현상이 나타났습니다.

### [원인 3] 미들웨어와 페이지의 세션 상태 동기화 충돌 (실제 발생했던 무한 루프)
* 과거 운영 서버에서 실제로 발생했던 **"Too many redirections (리다이렉션 수 초과)" 무한 루프의 진원지**입니다.
* 기존 미들웨어는 쿠키(세션)의 유저 상태만 보고 다음과 같이 작동했습니다:
  `if (user.status !== 'approved') return NextResponse.redirect(...)` -> `/pending`으로 리다이렉트
* 그러나 관리자가 사용자를 승인하여 DB 상태는 `'approved'`로 변경되었음에도 불구하고, 브라우저에 남아 있는 세션 쿠키 내의 `status`는 여전히 `'pending'`으로 남아 있는 **Stale(오래된) 상태 불일치**가 존재했습니다.
* 이 상태에서 사용자가 `/reserve`에 접근할 때 다음과 같은 **동기화 충돌 루프**가 발생했습니다:
  1. **미들웨어 검사**: 브라우저의 오래된 쿠키(`status: 'pending'`)를 보고 `/pending`으로 리다이렉트 시킴.
  2. **`/pending` 페이지 (서버 컴포넌트)**: 실시간 DB를 확인하여 유저가 이미 승인(`approved`)되었음을 알고, 예약 페이지 `/reserve`로 리다이렉트(`redirect('/reserve')`) 시킴.
  3. **재요청**: 브라우저는 다시 `/reserve`로 가지만, 쿠키는 여전히 `'pending'`이므로 미들웨어가 다시 `/pending`으로 보냄.
  4. 이 과정이 무한히 핑퐁처럼 반복되며 브라우저에서 **리다이렉션 수 제한 초과(20회)** 에러를 발생시켰습니다.

## 4. 새롭게 추가되는 강제 HTTPS 리다이렉션과 외부 프록시 헤더의 필요성

이번 개선 과정에서 보안을 한층 강화하기 위해 미들웨어 맨 위에 **HTTP로 들어온 요청을 HTTPS로 강제 리다이렉트하는 코드**를 추가하기로 하였습니다.
```typescript
const forwardedProto = request.headers.get('x-forwarded-proto');
if (forwardedProto === 'http') {
  const httpsUrl = new URL(request.url);
  httpsUrl.protocol = 'https:';
  return NextResponse.redirect(httpsUrl);
}
```
* 만약 이 코드가 들어가는데 **외부 프록시에서 `X-Forwarded-Proto` 헤더 설정을 해주지 않는다면**:
  * Next.js는 HTTPS로 들어오는 모든 정상적인 요청에 대해서도 `x-forwarded-proto`가 없거나 `http`로 수신되어, 무조건 `https://...`로 리다이렉트하라는 응답을 무한히 주는 **새로운 형태의 리다이렉트 루프**를 유발하게 됩니다.
* 따라서 코드에 HTTP -> HTTPS 강제 승격 로직을 탑재함과 동시에, **외부 프록시 프록시 설정에 `X-Forwarded-Proto` 헤더를 반드시 추가**해주어야만 이 보안 로직이 루프 없이 정상적으로 동작하게 됩니다.

## 3. 해결 방안 및 100% 해결 원리

이번 수정 사항(코드 개선 및 외부 프록시 프록시 헤더 설정 추가)을 통해 문제가 해결되는 기술적 원리입니다.

### ① 회원가입 후 자동 로그인 제거 및 로그인 화면 유도 (`/login?message=signup_complete`)
* **해결 원리**: 가입 성공 시 자동으로 세션 쿠키를 굽는 대신, 가입 성공 후 사용자를 로그인 페이지(`/login`)로 리다이렉트하여 수동 로그인을 유도하도록 아키텍처를 개선했습니다.
* **효과**:
  - 회원가입 API 수준에서 로그인 세션을 생성하지 않으므로 가입 직후의 WebKit 세션 I/O 레이스 컨디션이 **원천적으로 100% 방지**됩니다.
  - 로그인 쿠키는 로그인 화면에서 사용자가 아이디/비밀번호를 입력하고 로그인에 성공했을 때 비로소 생성됩니다. 정보 입력에 최소 수 초의 물리적 시간이 소요되므로, 세션 쿠키의 스토리지 영속화(I/O)가 완전히 마무리된 뒤 후속 대기 페이지 이동이 처리되어 어떠한 모바일 환경에서도 오작동 없이 작동합니다.
  - 가입이 안전하게 완료되었음을 알리는 성공 배너가 로그인 화면 상단에 명시적으로 노출되어 가입 후 대기 상태에 대한 UX 편의성이 향상됩니다.

### ② 미들웨어(`src/proxy.ts`)에서 실시간 유저 상태 검사 제거
* **해결 원리**: 미들웨어는 오직 로그인 유무(`!user`)만 판정하고, 세부적인 회원 상태 검사(승인 대기, 강퇴 등)는 실시간 DB 조회가 가능한 **Layout/Page 단**에서만 처리하도록 검증 역할을 나눴습니다.
* **효과**: 세션 쿠키의 `'pending'` 상태가 만료되거나 꼬이더라도 미들웨어가 개입하여 리다이렉트를 걸지 않으므로, DB 실시간 정보 기준의 안전한 라우팅만 수행되어 무한 리다이렉트 루프가 100% 차단됩니다.

### ③ 상대 경로 리다이렉트 적용 (`NextResponse` 수동 제어)
* **해결 원리**: 미들웨어 및 로그아웃 API 등에서 절대 경로 대신 **상대 경로(`Location: /login`)** 헤더를 가진 NextResponse를 수동으로 반환하도록 수정했습니다.
* **효과**: Next.js 서버가 외부 호스트나 프로토콜 정보를 몰라도 브라우저가 자체적으로 현재 접속 주소(`https://yeyak.example.or.kr`)를 기준으로 안전하게 주소를 해석하게 만듭니다.

### ④ 외부 프록시 역방향 프록시 헤더 설정 (`X-Forwarded-Proto` 등)
* **해결 원리**: 외부 프록시 프록시 장비에 `X-Forwarded-Proto: https`, `X-Forwarded-Host` 설정을 추가했습니다.
* **효과**: Next.js가 원래의 접속 프로토콜(HTTPS)과 도메인을 정상적으로 파싱하여, 서버 측에서 프로토콜 다운그레이드(HTTP 전환) 현상이 원천 차단됩니다.

### ⑤ 레이아웃 보호 강화 (Pending 사용자 편의 개선)
* **해결 원리**: `UserLayout` 및 `AdminLayout`에서 권한이 맞지 않을 때 무조건 세션을 파괴하고 로그아웃시키던 규칙 대신, 유저의 실시간 DB 상태가 `'pending'`인 경우 세션을 유지한 채 안전하게 `/pending` 대기 화면으로 돌려보내도록 분기 처리를 추가했습니다.

### ⑥ 모바일 키패드 최적화 및 전화번호 입력 제한 (숫자 입력 최적화)
* **해결 원리**: 회원가입, 개인 정보 수정, 관리자단 사용자 수정 페이지의 전화번호 인풋에 숫자 이외의 문자 입력을 방지하고 모바일 자판을 최적화했습니다.
  - `type="tel"`, `inputMode="numeric"`, `pattern="[0-9]*"` 및 `maxLength={11}` 속성을 적용하여 아이폰이나 갤럭시에서 숫자 전용 패드가 뜨도록 설계했습니다.
  - `onChange` 핸들러에서 입력된 비숫자 문자를 실시간으로 정규식(`replace(/[^0-9]/g, '')`) 필터링하여 순수 숫자만 들어가도록 강제했습니다.
* **효과**: 모바일 기기에서의 입력 오타율을 줄이고, 기기 키패드가 일반 문자 자판으로 불필요하게 튕기는 자판 전환 버그를 완전히 방지하여 모바일 입력 접근성을 향상시켰습니다.
